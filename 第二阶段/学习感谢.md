### A Git与GitHub的核心区别
Git是一款**本地版本控制工具**，它会在项目目录下生成`.git`文件夹，记录每一次代码修改的历史，让我们可以随时回退到某个版本、对比变更差异，不需要依赖网络就能管理代码。
而GitHub是基于Git的**远程代码托管平台**，它把本地的Git仓库同步到云端，既可以备份代码，也能让多个开发者在同一个项目上协作，是程序员之间共享代码、参与开源项目的重要载体。

一些感想，或者想法：之前我一直把Git和GitHub混为一谈，以为它们是同一个东西，现在才明白Git是“本地账本”，GitHub是“云端仓库”，这个认知差是我入门时最大的障碍。

### B Git基础操作与分支管理
Git的核心操作围绕“工作区-暂存区-本地仓库-远程仓库”展开：
- `git clone`：把远程仓库克隆到本地
- `git add`：把修改的文件加入暂存区
- `git commit`：把暂存区的变更提交到本地仓库，生成版本记录
- `git push`：把本地仓库的版本同步到远程仓库
- `git branch` / `git checkout`：创建和切换分支，让不同功能可以并行开发而不影响主代码

分支管理是Git的精髓，比如我们可以基于`main`分支创建`feat/功能名`分支开发新功能，完成后再合并回`main`，这样能有效避免代码冲突。

一些感想，或者想法：我在实操时遇到过“不是Git仓库”的错误，后来才明白必须进入包含`.git`的目录才能执行Git命令，这个细节让我对“目录结构”的重要性有了直观感受。

### C GitHub协作流程与PR提交
在GitHub上协作的标准流程是：
1.  Fork目标仓库到自己的账号下
2.  克隆Fork后的仓库到本地
3.  创建功能分支并完成开发
4.  提交变更并推送到远程分支
5.  在GitHub上发起Pull Request（PR），请求把自己的代码合并到原仓库
6.  等待原仓库维护者审核、评论、合并

PR机制不仅是代码提交的入口，更是代码 review、知识沉淀的重要环节，能保证项目代码的质量和一致性。

一些感想，或者想法：这次提交学习感想的实践，让我完整走了一遍从创建分支到发起PR的流程，深刻体会到“规范协作”比“写代码本身”更能保障项目的可持续性。
