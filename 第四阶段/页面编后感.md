# 🌟 Geek招新网页制作编后感

>从零到一，从本地到线上
—— 一次真正的“踩坑”成长记

这次完成 Geek 团队招新网页 的完整开发与部署，是我第一次 从 0 到 1 独立做出一个正式的前端页面。从最初脑子里只有“标签 + 样式”的模糊概念，到最终页面顺利上线、能被所有人访问，这中间走过的每一步都刻满了 “坑” 与 “爬出来” 的痕迹。

现在回看，那些调试到深夜的挫败、发现错误时的恍然大悟、以及最后看到页面完美呈现的狂喜，都成了我前端路上最宝贵的基石。

---

## 📐 一、页面搭建：从“我以为”到“我错了”

刚写 HTML 结构时，我天真地以为熟记几个常用标签和 CSS 属性就能顺利推进。结果真正动起手来，布局、定位、适配 每一个环节都给我上了生动一课。

### 🎨 1. 标题装饰的“捉迷藏”

制作 Introduction 板块时，我想给标题加一条波浪装饰线。本以为很简单——放一张背景图就行。
但图片要么不显示，要么尺寸巨大，要么位置偏移到九霄云外，更别提居中对齐了。

我反复检查路径、确认图片格式、调整 background-size / background-position，甚至怀疑过是不是图片本身有问题。最后沉下心来研究了 Flex 布局 与 背景图属性 的配合关系，才明白：

```css
.parent {
  display: flex;
  align-items: center;
}
.child-bg {
  background-image: url(...);
  background-size: contain;
  background-repeat: no-repeat;
}
```

装饰线终于乖乖地“贴”在了标题旁边。那一刻我真正理解了 父容器与子元素的定位关系——不是靠猜，而是靠逻辑。

### 🃏 2. About Us 板块的“三兄弟”居中难题

在 About Us 区域，Speech、Diversity、Time 三张图片需要水平排列并居中。起初卡片东倒西歪，图片大小不一，整个版面像被打翻的积木。

我尝试了 text-align: center 无效，后来才知道：

块级元素需要用 margin: 0 auto 来水平居中

同时给图片统一设置：

```css
img {
  display: block;
  width: 100%;
  max-width: 200px;
  margin: 0 auto;
}
```

再配合 Flex 的 justify-content: space-around 和 flex-wrap: wrap，三个“兄弟”终于规规矩矩地站成一排。

### ✨ 3. 四角装饰图的“越界捣乱”

为了增加页面设计感，我想在四个角落添加装饰图片。结果图片要么超出边界滚动条出现，要么遮挡文字，要么被其他元素压在底下。

这次我学会了 伪元素 + 绝对定位 的组合拳：

```css
.corner::before {
  content: "";
  position: absolute;
  top: 0;
  left: 0;
  width: 50px;
  height: 50px;
  background: url(decoration.png) no-repeat center/contain;
  z-index: 1;
  opacity: 0.8;
}
```

反复调试 top、left、right、bottom 后，装饰图终于安静地待在角落里，既不越界也不挡字。

### 📞 4. Contact Us 背景图的精准布阵

最后一个板块的背景图想要全幅覆盖。我折腾了 background-attachment: fixed、background-position: center、background-size: cover，还因为 z-index 没设好导致背景图把表单按钮盖住了。

最后通过给背景容器设置较低的 z-index，同时确保内部内容容器有独立的层叠上下文，才达到理想效果。

## 🧨 二、其他“惊喜”不断
```
问题 解决方案
固定导航栏遮挡内容 给主体内容加上 padding-top，数值等于导航栏高度
卡片动画失效 检查 transition 起始/结束状态，添加 overflow: hidden
图片拉伸变形 统一使用 object-fit: cover 并限制容器宽高
```

🚀 三、GitHub 部署：真正的“噩梦”才刚刚开始

本地页面看起来完美了，我以为上传到 GitHub Pages 就是几分钟的事。
没想到，部署环节才是最大的“坑王”。

### ❌ 1. 中文文件夹名引发的血案

我的图片文件夹命名为 图片素材，本地一切正常。上传后图片全部裂开。
查了半天发现：

GitHub Pages 对中文路径的解析极不稳定

我手动改为 images，并在所有代码中替换路径。刷新后还是不行——浏览器缓存了旧路径。最后清空缓存、用无痕模式，图片才重新出现。

### 📁 2. 多层嵌套的路径地狱

我把 CSS、图片、字体分别塞进 /assets/css/、/assets/images/ 等目录，结果有的 HTML 引用 href="style.css" 自然找不到。
不同二级目录下的文件引用公共资源时，相对路径各不相同，导致样式全丢。

解决方法： 统一改用 根目录相对路径（以 / 开头），并确保所有资源都放在仓库根目录下可访问的位置。

### 🔁 3. 代码路径不匹配的反复折磨

有一张背景图我引用了 ../images/bg.jpg，本地没问题，但 GitHub Pages 的根目录是仓库名，导致路径错位。
我用了 /项目名/images/bg.jpg 的绝对写法，换分支又失效。最终老老实实使用 相对路径，并手动验证每一个 src 和 href。

### 🧠 4. 浏览器缓存——看不见的敌人

每次修改后 push 到 GitHub，刷新自己的页面却还是旧样子。
一度怀疑 GitHub Pages 没更新，直到按下 Ctrl + F5 强制刷新才看到新版本。
后来养成了部署后清空缓存、甚至用不同设备验证的习惯。
